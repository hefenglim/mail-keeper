# Feature Specification: 大量信箱的效能與韌性（Bulk Resilience, R7）

**Feature Branch**: `005-bulk-resilience`

**Created**: 2026-06-24

**Status**: Draft

**Input**: User description: "大量信箱的效能與韌性（R7）—— token 過期/斷線自動續期重連並從斷點續做、消除同一流程內重複整夾讀取、暫時性錯誤退避重試、韌性門檻可由使用者設定、恢復期間仍顯示進度。"

## Clarifications

### Session 2026-06-24

- Q: 靜默續期不可行（refresh token 失效/被撤銷）時，系統該怎麼做？ → A: **乾淨停止** —— 印清楚的「需重新登入」訊息＋回報已完成/未完成數量；使用者重新登入後重跑同一份工作表續完（冪等，已搬走者自動跳過）。不在操作中途插入互動登入。
- Q: 「同一來源夾整夾標頭最多讀一次」時，哪一次是權威、TOCTOU 如何保證？ → A: 以「**檢查報告**」時讀的那一次為**權威快取**，執行階段重用、**不做第二次整夾掃描**；確認後到執行前消失的郵件，由搬移動作本身安全失敗回報（既有 stale-uid 處理，冪等）。
- Q: 自動恢復的範圍與續做細粒度（匯出 vs 分類）？ → A: 兩者**都**自動恢復；**分類（功能3）＝項目級續做**（只做剩餘未完成）；**匯出（功能1）＝重連後整批重抓**（唯讀、重跑安全，避免部分狀態 bug）。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 操作中途憑證過期或斷線仍自動完成（Priority: P1）

使用者對一個大型資料夾發起分類搬移（功能3），郵件數量多、耗時較長。操作進行到一半時，登入憑證（access token）到期、或網路連線中斷。使用者**不需要做任何事**：系統自動沿用既有授權取得新的憑證、重新建立連線，並從中斷的地方繼續處理尚未完成的郵件，直到全部完成，最後回報完整結果。

**Why this priority**: 這是 R7 的核心痛點。實測曾發生「28 封只成功 2 封、其餘全因連線中斷失敗」，使用者被迫手動重跑、對大信箱幾乎無法一次完成。修好它直接決定本功能對大量郵件是否可用。

**Independent Test**: 在「處理到中途憑證過期 / 連線中斷」的情境下發起分類搬移，全程不人工介入，驗證所有應搬移的郵件最終都搬移完成、無重複、無遺漏，並回報為成功完成（而非中止）。

**Acceptance Scenarios**:

1. **Given** 來源資料夾有大量待搬移郵件、且憑證將於操作中途到期，**When** 使用者啟動搬移並不做任何介入，**Then** 系統自動續期＋重連並完成全部待搬移郵件，最終回報全部成功、無重複搬移、無遺漏。
2. **Given** 操作進行中網路短暫中斷後恢復，**When** 系統偵測到連線中斷，**Then** 系統重新連線並從未完成處續做，不需使用者重新登入。
3. **Given** 操作因連線問題恢復後繼續，**When** 某封郵件在中斷期間已被他處搬走或刪除，**Then** 該封被如實標示為「執行時已不存在」而非重複搬移或崩潰（維持冪等）。
4. **Given** 既有授權已無法靜默續期（例如授權已被撤銷或長期未用而失效），**When** 系統嘗試自動續期失敗，**Then** 系統乾淨停止、清楚告知需要重新登入、並回報已完成與尚未完成的數量（不靜默吞、不假裝成功）。

---

### User Story 2 - 同一操作不重複下載整夾標頭（Priority: P2）

使用者對大型資料夾執行分類搬移。系統在「檢查報告」與「實際執行」兩個階段，對**同一個來源資料夾**只讀取一次整夾標頭，不重複下載，縮短大信箱的等待時間與流量。

**Why this priority**: 目前同一分類流程會對同一來源資料夾讀取兩次整夾標頭（報告一次、執行一次），對大信箱是可省的重複下載。屬效能優化，不影響正確性，故次於 P1。

**Independent Test**: 對一個來源資料夾執行一次完整分類搬移，量測該資料夾整夾標頭的讀取次數，驗證為 1 次。

**Acceptance Scenarios**:

1. **Given** 一份指向單一來源資料夾的工作表，**When** 使用者完成「檢查報告 → 確認 → 執行」整個分類流程，**Then** 該來源資料夾的整夾標頭在整個流程中只被讀取一次。
2. **Given** 報告與執行之間相隔一段時間、其間某封已被他處搬走，**When** 系統執行搬移（以報告快取為準、不重讀整夾），**Then** 該封的搬移動作安全失敗並被如實回報（既有 stale-uid 處理），不誤搬、不崩潰。

---

### User Story 3 - 暫時性錯誤退避重試且門檻可設定（Priority: P3）

面對暫時性的網路抖動，系統以有限次數的退避重試吸收，而非一遇到失敗就立刻計入「連續失敗、放棄」。使用者可在設定中調整韌性相關門檻（例如連續失敗上限、重試次數），不需修改程式。

**Why this priority**: 提升健壯度與可調性，但在 P1 自動恢復到位後屬加值；預設值已能運作，故優先級最低。

**Independent Test**: 注入一個「短暫失敗後即恢復」的情境，驗證操作能吸收該抖動並繼續完成；調整設定中的門檻值，驗證行為依設定改變。

**Acceptance Scenarios**:

1. **Given** 一封郵件的搬移遭遇一次短暫失敗、隨即可恢復，**When** 系統重試，**Then** 該封最終搬移成功，且不因該次抖動就觸發整體放棄。
2. **Given** 使用者在設定中調整了「連續失敗上限」，**When** 連續失敗達到新的門檻，**Then** 系統依新門檻決定是否停止（行為隨設定改變）。
3. **Given** 設定缺漏或無效，**When** 系統啟動，**Then** 以安全的內建預設值運作，不崩潰。

---

### Edge Cases

- **靜默續期不可行**：既有授權失效/被撤銷，無法不互動續期 → 乾淨停止、明確提示需重新登入、回報已完成/未完成數量（US1 場景4）。
- **連線反覆中斷（伺服器持續不可用）**：重連/重試在有限次數內仍失敗 → 在合理上限後停止，清楚說明剩餘未處理數量與下一步，而非無止境重試或刷同樣錯誤。
- **恢復期間使用者中斷（如 Ctrl-C）**：乾淨結束、不留下半完成的損毀狀態（已完成的搬移保持有效、不重複）。
- **中斷期間資料變動**：報告後、執行/恢復前，郵件被他處搬走或刪除 → 如實標示、不重搬、不崩潰（冪等）。
- **匯出（功能1）長時間下載中斷**：唯讀的標頭下載若中途斷線，重連後整批重抓，最終輸出完整、每列 UID 不為空。
- **恢復過程的可見性**：重連/續期/續做期間，使用者仍看得到進度或狀態，不會誤判當機。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統 MUST 在大量郵件操作（功能3 分類搬移、功能1 匯出）進行期間，偵測「憑證過期」與「連線中斷」這兩類中斷。
- **FR-002**: 偵測到中斷時，系統 MUST 嘗試**沿用既有授權、不需使用者重新互動登入**取得新憑證並重新建立連線。
- **FR-003**: 重連成功後，系統 MUST 從中斷處繼續直到全部完成，且不重複處理已完成的項目、不遺漏未完成的項目。續做細粒度依操作而定：**分類（功能3）為項目級續做**（只處理剩餘未完成的搬移）；**匯出（功能1）為重連後整批重抓**（唯讀、重跑安全）。
- **FR-004**: 當靜默續期不可行（授權失效/撤銷）時，系統 MUST 乾淨停止、明確告知使用者需重新登入，並回報「已完成」與「尚未完成」的數量；MUST NOT 靜默吞錯或誤報成功。
- **FR-005**: 系統 MUST 對暫時性錯誤施以**有限次數的退避重試**；僅當重試仍持續失敗（疑似真正中斷）時，才計入放棄條件。
- **FR-006**: 系統 MUST 在連線反覆失敗達到合理上限時停止，並清楚回報剩餘未處理數量與建議下一步；MUST NOT 無止境重試。
- **FR-007**: 系統 MUST 在同一次分類流程中，對任一來源資料夾的整夾標頭**最多讀取一次**：以「檢查報告」階段所讀為**權威快取**，執行階段重用、**不做第二次整夾掃描**。確認後到執行前消失的郵件由搬移動作本身安全失敗回報（既有 stale-uid 處理，冪等），不因省略重讀而誤搬或崩潰。
- **FR-008**: 韌性相關門檻（至少：連續失敗上限、重試次數）MUST 可由使用者於設定調整；缺漏或無效時 MUST 以安全內建預設值運作、不崩潰。
- **FR-009**: 系統 MUST 在操作全程（含續期、重連、續做期間）提供進度或狀態提示，讓使用者不致誤判當機。
- **FR-010**: 所有恢復行為 MUST 維持冪等：重連後續做、或使用者事後重跑同一份工作表，皆不得造成重複搬移或資料遺失。
- **FR-011**: 破壞性動作 MUST 維持現有安全預設（功能3 預設 dry-run，確認後才搬移）；恢復機制不得繞過確認或擴大刪除範圍。
- **FR-012**: 憑證、token 等敏感資訊 MUST NOT 因恢復/重連流程而被記錄或輸出。
- **FR-013**: 進度狀態條的啟動 MUST 依迴圈性質而定：**涉及網路 in/out 的迴圈一律顯示狀態條**（不設件數門檻——每次網路往返本就緩慢、應即時回饋，即使件數 ≤ 30）；**純 CPU/記憶體內的迴圈僅在件數 > 30 時顯示**（避免瞬間完成的迴圈閃爍）。其餘既有行為不變：僅互動 TTY 顯示、非 TTY 零輸出、不污染資料輸出、輸出例外被吞不崩潰。

### Key Entities *(include if feature involves data)*

- **操作工作清單（Work List）**：一次操作中所有待處理項目及其狀態（待處理／已完成／失敗／已不存在）。用於支援「從斷點續做」與「不重複、不遺漏」。
- **韌性設定（Resilience Settings）**：使用者可調整的門檻集合（連續失敗上限、重試次數等），具安全預設值。
- **連線工作階段（Session）**：與信箱的連線及其有效性；中斷後可在不需互動登入的前提下重建。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在「憑證於操作中途過期」的情境下，分類搬移最終完成 **100%** 應搬移的郵件，**0** 重複、**0** 遺漏，且**不需使用者手動介入**。
- **SC-002**: 在「操作中途連線中斷一次後恢復」的情境下，操作在**不需使用者重新登入**的前提下恢復並完成剩餘工作。
- **SC-003**: 同一次分類流程中，任一來源資料夾的整夾標頭讀取次數為 **1**（目前為 2）。
- **SC-004**: 當靜默續期不可行時，**100%** 的此類情境以「乾淨停止＋明確需重新登入提示＋正確的已完成/未完成數量」收場，無任何靜默失敗或誤報成功。
- **SC-005**: 重跑同一份工作表（或重連後續做）造成的重複搬移與資料遺失次數為 **0**（冪等）。
- **SC-006**: 操作全程（含恢復期間）皆有可見的進度或狀態，使用者回報「不像當機」。
- **SC-007**: 任一網路讀取/搬移迴圈（即使件數 ≤ 30）在互動 TTY 下皆顯示狀態條；純 CPU 迴圈於 ≤ 30 件時不顯示（無閃爍）。

## Assumptions

- **沿用既有 OAuth 授權即可靜默續期**：在 refresh token 仍有效的前提下可不互動取得新 access token；若已失效則屬 US1 場景4 的「乾淨停止」路徑。
- **預設行為為自動恢復、非互動**：與專案「非互動安全」原則一致；不在操作中途彈出互動登入，而是在無法續期時乾淨停止並提示。沿用既有「dry-run 預設、確認後搬移」的安全模型。
- **本期不含超大信箱的分頁/串流**：UID 範圍開窗、分頁、邊抓邊處理以降低記憶體，留待後續；本期聚焦恢復、去重讀、退避重試與可設定門檻。
- **韌性門檻沿用既有設定機制**：放入既有的工作目錄設定（與其他非機密設定同處），具安全預設值。
- **範圍外**：切換到 Microsoft Graph 後端（R3）、整理規則外部化（R5）、結構化 logging（R6）、LLM 自動串接三功能。
- **後端介面不變**：恢復/重連能力以現有後端抽象提供；上層（分類引擎、cli）不因更換後端而需改寫。
