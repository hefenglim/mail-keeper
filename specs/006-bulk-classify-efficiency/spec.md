## User Scenarios & Testing *(mandatory)*

### User Story 1 - 大型信箱的分類檢查報告快速且省流量 (Priority: P1)

使用者對一個有數千～上萬封郵件的來源資料夾執行「依 CSV 分類」。在真正搬移前，系統會先產出一份「檢查報告」，逐列標示該列是可搬移（candidate）、略過（skip）或不可行（infeasible，例如來源郵件已不存在）。使用者期望這份報告**很快**出現、**不耗費大量流量**，因為它只是「確認來源郵件還在不在」，而不是要閱讀郵件內容。

**Why this priority**: 這是本期唯一也是最大的效能價值。實測顯示，現況為了「確認存在性」竟下載整夾所有郵件的完整標頭再全部丟棄（10,000 封 ≈ 200 批網路往返、下傳 ~1.63 MB）。消除此浪費能大幅縮短大型信箱的報告時間、降低流量，並縮小操作被存取權杖（token）過期打斷的時間窗。此故事獨立交付明確價值，且不更動搬移行為。

**Independent Test**: 準備一個大型來源資料夾與一份對該夾的分類工作表，執行到「檢查報告」階段；驗證 (1) 報告逐列判定與優化前完全一致、(2) 該來源夾沒有發生「整夾完整標頭」抓取、(3) 報告階段的下載資料量大幅下降。

**Acceptance Scenarios**:

1. **Given** 一個含 10,000 封郵件的來源資料夾與一份引用其中部分 UID 的分類工作表，**When** 執行分類並產出檢查報告，**Then** 系統未對該資料夾下載/解析整夾郵件的完整標頭（主旨/寄件者/收件者/日期），且每列的 skip/candidate/infeasible 判定與原因，與優化前對相同輸入逐列一致。
2. **Given** 同一份分類工作表的多列來源郵件都在同一個來源資料夾，**When** 產出檢查報告，**Then** 該資料夾的「現存項目查詢」在整個分類流程中最多發生一次（報告與後續執行共用、不重複查詢）。
3. **Given** 工作表中有一列的來源 UID 實際上已不存在於來源夾，**When** 產出檢查報告，**Then** 該列被標為 infeasible（原因：來源郵件不存在），與優化前一致。
4. **Given** 來源夾中有一封郵件已被標記 \Deleted 但尚未 expunge，且工作表有一列引用其 UID，**When** 產出檢查報告，**Then** 該列被視為來源存在（candidate），與現況一致（存在性涵蓋已標刪未清除者）。

---

### User Story 2 - 內容讀取功能與安全行為不受影響 (Priority: P2)

使用者仍需使用「匯出工作表」（功能1）與「列出標題」（功能2）等需要**真正讀取郵件標題內容**的功能，且分類的搬移執行行為（含進度、結果、早停、dry-run 預設）維持不變。使用者期望本次效能優化**只影響**分類的存在性檢查、**不影響**其他任何路徑，也不外洩任何祕密。

**Why this priority**: 這是回歸防線。本期只動「分類的存在性來源」，必須確保不誤傷需要內容的路徑（功能1/2）與搬移執行流程。

**Independent Test**: 對同一信箱執行匯出/列標題，驗證輸出（含主旨/寄件者解碼）與優化前一致；對同一工作表執行搬移，驗證搬移結果與逐列輸出順序與優化前一致；確認破壞性動作仍預設 dry-run、輸出無任何祕密。

**Acceptance Scenarios**:

1. **Given** 一個含多種編碼主旨（ASCII/CJK/emoji/encoded-word）的信箱，**When** 執行匯出工作表或列出標題，**Then** 輸出的主旨/寄件者等內容與優化前逐字一致（這些路徑仍取得完整標頭）。
2. **Given** 任何分類操作，**When** 未明確加上執行旗標，**Then** 仍預設為 dry-run（不變更郵件），且任何輸出/錯誤訊息皆不含 token 或祕密。
3. **Given** 一份分類工作表，**When** 確認後執行搬移，**Then** 最終搬移結果集合、逐列成功/失敗與其順序、進度回報，與優化前一致（本期不改動執行/搬移路徑）。

### Edge Cases

- **空來源夾**：來源資料夾存在但沒有郵件 → 該夾現存集合為空；引用其 UID 的列被標 infeasible（不存在），與優化前一致。
- **來源夾不存在**：工作表列的來源夾不在信箱中 → 標 infeasible（來源資料夾不存在），不觸發任何現存查詢。
- **已標刪未清除的來源郵件**：仍視為「存在」（與現況一致，存在性涵蓋已標 \Deleted 未 expunge 者）；引用其 UID 的列可為 candidate，搬移行為不變。
- **同夾多列、重複 UID**：現存查詢每夾仍只一次；判定逐列正確。
- **報告時存在、執行時已不存在**：來源 UID 在報告階段判為存在、但執行搬移時已被他處移走/刪除 → 仍如實回報該列失敗（行為不變）。
- **連線中斷／權杖過期於現存查詢期間**：透明重連後完成查詢；結果不重複、不遺漏（與既有重連語意相容）。
- **超大來源夾（>10,000 封）**：現存查詢仍為單次最小化查詢，不退化為整夾內容分批下載。
- **內容讀取路徑不得被誤最小化**：匯出/列標題仍須取得完整標頭——存在性最小化僅限分類的存在性判斷。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 分類「檢查報告」在判斷某列來源郵件是否存在時，MUST 僅取得「來源資料夾目前現存項目的識別資訊（UID 集合）」，且 MUST NOT 為了存在性判斷而下載或解析該資料夾郵件的完整標頭內容（主旨／寄件者／收件者／日期）。「現存」的認定與現況一致：涵蓋信箱中尚未被清除(expunge)的所有郵件，**包含已標記 \Deleted 但未清除者**。
- **FR-002**: 同一次分類流程中，任一來源資料夾的「現存項目查詢」MUST 最多執行一次，並由檢查報告階段與執行階段共用（不得重複整夾查詢）。
- **FR-003**: 對相同輸入工作表，「檢查報告」對每一列的判定（skip／candidate／infeasible）與其原因文字，MUST 與優化前逐列完全一致。
- **FR-004**: 檢查報告的「列出順序」MUST 維持與輸入工作表一致（本期不改動順序）。
- **FR-005**: 系統在分類全程（含恢復／重連期間）MUST 持續顯示進度，且進度回報語意（已完成數／總數）不退化。特別是「來源夾存在性查詢」步驟 MUST 仍呈現逐項式（determinate）進度——其總數反映該來源夾的郵件數並推進至完成，**不得退化為無回饋**。
- **FR-006**: 需要郵件「內容」的既有功能（匯出工作表、列出標題）MUST 維持行為不變，繼續取得完整標頭內容；分類的**搬移執行路徑**（結果集合、逐列輸出順序、早停、dry-run 預設）MUST 維持不變（本期不改動）。
- **FR-007**: 「取得來源資料夾現存 UID 集合」的能力 MUST 以後端中立方式提供（透過既有後端抽象新增能力，由各後端各自實作）；上層分類／CLI 邏輯 MUST NOT 特例化任一後端，且任何後端協定細節 MUST NOT 跨越後端邊界外洩到上層。
- **FR-008**: 本變更 MUST NOT 記錄或輸出任何存取權杖／祕密，且 MUST NOT 變更 OAuth 授權範圍或重新引入密碼登入。
- **FR-009**: 「現存項目查詢」MUST 與既有的透明重連／有界重試行為相容——連線中斷時重連後仍能完成查詢，且不產生重複或遺漏的判定。

### Key Entities *(include if feature involves data)*

- **來源夾現存集合（Source Folder Presence Set）**：某一來源資料夾「目前存在哪些郵件」的識別碼（UID）集合，用於判斷工作表各列的來源郵件是否仍存在。涵蓋信箱中尚未被 expunge 的所有郵件（含已標 \Deleted 未清除者），與現況一致。屬輕量識別資訊，不含郵件內容。存於既有 `ClassifyCache.source_uids`，一次分類流程每夾最多取一次。
- **檢查報告項目（Report Item）**：工作表每一列的判定結果（skip／candidate／infeasible 與原因）。其逐列內容（FR-003）與列出順序（FR-004）為正確性比對基準。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 對一個 10,000 封的來源資料夾執行分類檢查報告，該資料夾發生的「整夾完整標頭抓取」次數為 **0**（以可觀測的操作日誌驗證）。
- **SC-002**: 對相同輸入工作表，優化前後的檢查報告逐列判定（skip／candidate／infeasible＋原因）一致率為 **100%**（含「已標刪未清除郵件視為存在」之情境）。
- **SC-003**: 對 10,000 封的來源資料夾，檢查報告階段自伺服器下載的資料量相較現況下降 **≥ 90%**（由「整夾標頭」量級降為「UID 清單」量級）。
- **SC-004**: 優化前後，分類搬移的最終結果集合（成功搬移的郵件與其目標夾、每列成功／失敗、逐列輸出順序）一致率為 **100%**（本期不改動搬移路徑）。
- **SC-005**: 既有匯出工作表／列出標題功能的輸出，優化前後一致率為 **100%**（無回歸）。
- **SC-006**: 分類全程（含恢復／重連期間）皆有進度回饋，無「無回饋、看似當機」的等待區段；其中存在性查詢步驟呈現逐項式（determinate）進度並推進至完成。

## Assumptions

- 後端能以單一輕量查詢回傳「某資料夾現存郵件的識別碼集合」（IMAP 提供等價的搜尋能力；離線測試用的 `FakeBackend` 亦可提供相同能力）。
- 既有「每個來源夾在一次分類流程只讀一次」的快取精神（現由 `ClassifyCache` 體現）延續到「現存項目查詢」。
- 分類由 CSV 的 current／target 欄位驅動，**不以郵件內容**決定分類——因此分類存在性判斷不需要郵件內容，可安全最小化。
- 「下載資料量下降 ≥ 90%」以引擎可觀測的傳輸量（bytes）或等價指標衡量；絕對值依信箱大小而定，比較採同一情景前後對照。
- 「逐項式進度」對單次查詢而言，其總數取自來源夾郵件數並推進至完成；少次往返時狀態條會快速填滿（真實速度，非缺陷），不注入人工延遲。
- 本期**不改動**分類的搬移執行路徑（排序/批次/早停皆不動）；P4 候選分組與 P2/P3/P5/P6/P7、backlog C1/C2 均為後續階段。

## Dependencies

- 既有後端抽象（`MailBackend`）與分類流程的共用快取（`ClassifyCache`）。

- IMAP 模擬器引擎（`tests/imap_server.py::ImapServer` + `SimIMAP4_SSL`）作為唯一核可的跨 seam 測試與量測工具（正常＋異常情境、操作日誌驗證）。
