# Feature Specification: 啟動選單與 CSV 郵件匯出／分類

**Feature Branch**: `003-csv-classify`

**Created**: 2026-06-21

**Status**: Draft

**Input**: 使用者描述：「啟動 mailkeeper 出現選單，選單功能有讀取 inbox 所有 email 的 UID、標題、寄件者、位於哪個資料夾等有用的資訊，然後輸出成 csv 檔案；選單另一個功能則是輸出 csv 列出所有資料夾的資訊；第三個功能為將郵件依編輯後的 csv 自動分類到資料夾內。」

**Refinement (2026-06-21)**：三個功能串成一條工作流 —— 功能1（「匯出資料夾的所有電子郵件列表」）先讓使用者**選擇一個資料夾**，再把該資料夾的郵件匯出成一份「可編輯的分類工作表」CSV；每列包含足以讓 **AI / LLM Agent 進行分類**的標頭類資訊（寄件者、收件者、主旨、日期、所在資料夾），以及一個可編輯的 `target_folder` 欄位（不含郵件內文）。使用者或一個外部 AI Agent 參考功能2 匯出的「所有資料夾清單」，在這份 CSV 上為每封郵件決定並填入 `target_folder`。最後功能3 匯入這份編輯過的 CSV，對「資料夾有變動」的列做可行性初步檢查並產生報告，使用者確認無誤後才依 CSV 執行搬移。功能1 與功能3 使用**同一份 CSV**；MailKeeper 本身不執行 LLM，只負責產出足夠資訊的 CSV 並消費填好的 CSV。

**Phasing（本期範圍）**：本規格為**先驗證階段** —— 三個功能各自獨立、以**手動流程**運作：CSV 由**人＋AI 在工具外協作編輯**，功能3 匯入後做檢查報告、全數通過並經使用者確認，才執行資料夾搬移。待此手動流程驗證可靠後，**未來**再考慮導入 LLM 底層，把三個功能自動串接成端到端流程 —— 該自動化**不在本期範圍**。

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 選擇資料夾並匯出可供 AI 分類的工作表（功能1）(Priority: P1)

使用者啟動 MailKeeper 後從選單選「匯出資料夾的所有電子郵件列表」。工具先讓使用者**從信箱資料夾中選擇一個**，再把該資料夾的每一封郵件寫成一份 CSV。每列依**固定欄位順序**包含 `uid`、`current_folder`、`target_folder`（空白、可編輯）、`date`、`from`、`to`、`subject`，皆為標頭類資訊、不讀取郵件內文。使用者或外部 AI Agent 之後即可在這份 CSV 上，為每封郵件決定並填入新資料夾。

**Why this priority**：這份 CSV 是整條分類工作流的核心載體（功能1 產出、人／AI 編輯、功能3 消費）。它本身是唯讀匯出、零風險，且其欄位是否「足以分類」直接決定整個流程的成效。

**Independent Test**：以含多資料夾、各含中英文郵件的假信箱選功能1、指定某資料夾，確認 CSV 每列依序含 `uid, current_folder, target_folder, date, from, to, subject`（`target_folder` 空白），且能被標準 CSV 工具正確開啟。

**Acceptance Scenarios**：

1. **假設** 信箱有多個資料夾，**當** 使用者選「匯出資料夾的所有電子郵件列表」，**則** 工具先呈現可選資料夾、待使用者選定後，匯出該資料夾郵件的 CSV，過程不變更信箱。
2. **假設** 已選定資料夾且其中有郵件，**當** 匯出，**則** 產生一份 UTF-8 CSV，每列依序含 `uid`、`current_folder`、`target_folder`（空白可編輯）、`date`、`from`、`to`、`subject`。
3. **假設** 某封郵件主旨或寄件者含逗號／引號／換行，**當** 匯出時，**則** 該欄以標準 CSV 跳脫處理，開啟後欄位不錯位。
4. **假設** 所選資料夾為空，**當** 匯出時，**則** 產生只有表頭的 CSV，提示 0 封、無錯誤。

---

### User Story 2 — 匯出所有資料夾清單作為編輯參考（功能2）(Priority: P1)

使用者從選單選「匯出資料夾清單」。工具列舉信箱中的所有資料夾，輸出一份 CSV 列出每個資料夾名稱（本期不含郵件數）。使用者據此得知有哪些有效的資料夾名稱，以便在功能1 的工作表中正確填寫 `target_folder`。

**Why this priority**：提供有效目標資料夾的權威清單，避免使用者在工作表中填入不存在或拼錯的資料夾名，是讓功能3 檢查能通過的前置。

**Independent Test**：以含多個資料夾的假信箱選功能2，確認 CSV 列出所有資料夾名稱（涵蓋 INBOX 與自訂資料夾）。

**Acceptance Scenarios**：

1. **假設** 信箱有 INBOX 與數個自訂資料夾，**當** 使用者選「匯出資料夾清單」，**則** CSV 列出所有資料夾名稱，過程不變更信箱。
2. **假設** 信箱結構有巢狀資料夾，**當** 匯出時，**則** 以可辨識的完整名稱列出每個資料夾。

---

### User Story 3 — 匯入編輯後的 CSV，檢查、確認、搬移（功能3）(Priority: P2)

使用者把在功能1 工作表上編輯好（為部分郵件填了 `target_folder`）的 CSV，從選單選「依工作表分類」匯入。工具逐列比較 `target_folder` 與 `current_folder`：**只有有變動（`target_folder` 已填且不等於 `current_folder`）的列才視為搬移候選**。工具對每個候選做**初步檢查**（目標資料夾是否合理、是否存在或可建立、來源郵件是否存在、`uid` 是否有效），產生一份**檢查報告**列出「將把哪一封從哪搬到哪」與不可行項，且全程不變更信箱。使用者**確認無誤**後，工具才真正逐封搬移，並回報結果。

**Why this priority**：這是最有價值但具破壞性的步驟，必須建立在工作表格式與資料夾能力之上，並以「初步檢查報告＋明確確認」保護。

**Independent Test**：以假後端注入一份編輯過的 CSV（含有變動且可行的列、未變動的列、以及目標不合理的列），確認檢查報告正確分類三者、確認前未搬任何郵件；確認後僅可行的變動列被搬到正確資料夾。

**Acceptance Scenarios**：

1. **假設** 一份 CSV 中部分列已填 `target_folder` 且與 `current_folder` 不同，**當** 使用者選功能3，**則** 工具產生初步檢查報告列出每個搬移候選「從 current_folder 搬到 target_folder」及其可行性，且**確認前不變更任何郵件**。
2. **假設** 某些列的 `target_folder` 為空白或等於 `current_folder`，**當** 檢查時，**則** 這些列被視為「無變動」、不納入搬移、不做任何動作。
3. **假設** 某列的 `target_folder` 不合理（無法建立或無效）或來源郵件不存在，**當** 檢查時，**則** 該列在報告中標示為不可行並排除，不會被搬移。
4. **假設** 使用者於報告後明確確認，**當** 執行，**則** 每個可行的變動列其郵件被搬至 `target_folder`，並回報成功筆數。
5. **假設** 使用者於報告後取消，**當** 取消，**則** 信箱完全不變更、以非破壞性結束。
6. **假設** 執行中途部分搬移失敗，**當** 結束，**則** 回報哪些成功／失敗，已搬的維持已搬、未搬的維持原樣，信箱不處於不明狀態。

---

### Edge Cases

- 非互動環境（無終端機／管線／CI）啟動選單：必須安全處理（以參數選功能或顯示用法後明確結束），不得卡死。
- CSV 缺檔、路徑不可讀寫、格式錯誤、缺必要欄位（`uid`／`current_folder`／`target_folder`）、表頭不符。
- `target_folder` 填了不存在的資料夾：依設定自動建立或標為不可行（檢查報告說明）。
- 來源 `uid` 不存在、或已被同次執行較早的列搬走（順序相依；執行時將該列列為失敗，見 FR-011）。
- 重複列、或 `target_folder` 等於 `current_folder`（視為無變動）。
- 使用者手動改動了識別欄（`uid`／`current_folder`）導致對不到來源郵件。
- 標題／寄件者含逗號、引號、換行、各國語文與 emoji。
- 大量郵件與多資料夾下的匯出與搬移（效能／分批屬未來 R7，**本期沿用逐封 fetch、不在本期範圍**）。
- 匯出檔已存在：覆寫或另存的處理。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**：啟動時系統 MUST 顯示互動選單，至少提供：功能1（選擇資料夾並匯出該資料夾郵件的分類工作表 CSV）、功能2（匯出資料夾清單 CSV）、功能3（依工作表 CSV 分類）、離開。
- **FR-002**：非互動環境（無 TTY）下 MUST 安全處理：(a) 提供子指令時直接執行該功能；(b) 無子指令時印出用法並以非零碼結束。兩者皆不得卡死或崩潰。
- **FR-003**：功能1 MUST 先讓使用者選擇一個資料夾，再將該資料夾每一封郵件輸出為 CSV（UTF-8、含表頭）；每列 MUST 依**固定欄位順序**輸出 `uid`、`current_folder`、`target_folder`（預設空白、可由人或 AI Agent 編輯）、`date`、`from`、`to`、`subject`，皆為標頭類、不含郵件內文。
- **FR-004**：功能2 MUST 列舉信箱所有資料夾並輸出為 CSV（每列一個資料夾名稱），作為填寫 `target_folder` 的有效名稱參考。（郵件數 `count` 不在本期範圍、列為未來。）
- **FR-005**：所有 CSV 匯出 MUST 對含逗號、引號、換行與非 ASCII 的欄位正確跳脫，使輸出可被標準 CSV 工具正確解析、不錯位。
- **FR-006**：功能3 MUST 匯入功能1 格式的 CSV，並以 (`current_folder`, `uid`) 唯一識別每一封來源郵件、以 `target_folder` 為目標資料夾。
- **FR-007**：功能3 MUST 只把「`target_folder` 已填且不等於 `current_folder`」的列視為搬移候選；`target_folder` 空白或等於 `current_folder` 的列 MUST 視為無變動、不做任何動作。
- **FR-008**：功能3 MUST 在搬移前先產生初步檢查報告：逐一驗證每個候選（目標資料夾合理且存在或可建立、來源郵件存在、`uid` 有效），列出將搬移項與不可行項，過程不得變更任何郵件。
- **FR-009**：功能3 MUST 僅在使用者明確確認且檢查通過後才實際搬移；確認前不得變動任何郵件。
- **FR-010**：功能3 對不可行或格式錯誤的列 MUST 清楚標示並排除，不得靜默誤搬；其餘可行的變動列仍可在確認後執行。
- **FR-011**：功能3 的搬移 MUST 盡量具冪等性；部分失敗時 MUST 回報每列成功／失敗，且不使信箱處於不明狀態。執行時若某列來源 UID 已不存在（例如已被同次較早的列搬走），MUST 將該列回報為失敗、不得崩潰。
- **FR-012**：CSV 匯出與匯入的檔案路徑 MUST 可由使用者指定（否則用工作目錄下的預設檔名）；路徑不可讀寫時 MUST 以乾淨訊息報錯、不崩潰。
- **FR-013**：系統 MUST 能列舉信箱資料夾、讀取指定資料夾郵件的標題類資訊（主旨／寄件者／收件者／日期）、（必要時）建立目標資料夾，並在資料夾之間搬移郵件（功能1／2／3 所需能力）。
- **FR-014**：所有輸出與錯誤 MUST 沿用既有保證 —— 編碼安全、永不因已知失敗崩潰或卡死、不外洩 token、已知失敗以乾淨訊息與非零碼結束。
- **FR-015**：MUST 維持後端隔離 —— 列舉資料夾、跨資料夾搬移等能力以 `MailBackend` 介面方法提供，`imaplib` 細節不外洩過界、且僅出現在 `imap_client.py`。
- **FR-016**：功能1／2 匯出時，若目標檔已存在 MUST 預設覆寫（未來可加另存／詢問選項）。

### Key Entities *(include if feature involves data)*

- **選單 (Menu)**：啟動後呈現的功能選項集合（功能1／2／3／離開）。
- **分類工作表列 (Classification Worksheet Row)**：固定欄位順序為 `uid`、`current_folder`、`target_folder`（可編輯）、`date`、`from`、`to`、`subject`（全英文表頭）。此為功能1 產出、人／AI 編輯、功能3 消費的**同一份檔案**；唯一鍵為 (`current_folder`, `uid`)。僅含標頭類欄位、不含郵件內文。
- **資料夾清單列 (Folder Reference Row)**：資料夾名稱（本期僅此欄；`count` 郵件數列為未來）。功能2 產出，供填寫 `target_folder` 參考。
- **資料夾 (Folder)**：信箱中的郵件資料夾，可被列舉、可作為搬移目標（必要時建立）。
- **分類者 (Classifier，外部)**：閱讀工作表 CSV 並填寫 `target_folder` 的人或 AI / LLM Agent。MailKeeper 不執行 LLM，僅負責產出足夠資訊的 CSV 與消費填好的 CSV。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**：使用者啟動後可在選單中以最多 2 步選定任一功能。
- **SC-002**：對含至少 5 種語文／特殊字元的信箱，功能1／2 匯出的 CSV 100% 能被標準 CSV／試算表工具正確開啟、欄位不錯位、零崩潰。
- **SC-003**：功能3 在使用者確認前 100% 不變更任何郵件；初步檢查報告正確區分「將搬移／無變動／不可行」三類。
- **SC-004**：依編輯後 CSV 確認後，100% 可行的變動列其郵件被搬至正確 `target_folder`；任何失敗列被明確回報。
- **SC-005**：`target_folder` 空白或等於 `current_folder` 的列，搬移數為 0。
- **SC-006**：任何 CSV 格式錯誤、缺檔或路徑問題皆以乾淨訊息處理，零未捕捉崩潰、零卡死。
- **SC-007**：本功能的自動化測試 100% 可離線執行（注入 FakeBackend），不需網路。

## Assumptions

- **同一份 CSV 貫穿工作流**：功能1 產出分類工作表、使用者編輯 `target_folder`、功能3 消費；欄位相容。
- **郵件識別鍵**：採 (`current_folder`, `uid`)（IMAP UID 僅於單一資料夾內唯一）。
- **功能1 範圍**：由使用者**選擇單一資料夾**後匯出該資料夾的郵件（要處理多個資料夾就分別匯出、編輯、匯入）。`current_folder` 即所選資料夾。
- **AI 分類消費者**：工作表 CSV 的欄位以「供外部 AI / LLM Agent 有效分類」為設計目標；MailKeeper 僅產出與消費 CSV、不內建 LLM，`target_folder` 由人或 Agent 填寫。
- **不讀取內文**：經決議，功能1 的 CSV 只輸出標頭類欄位（`uid`／所在資料夾／寄件者／收件者／主旨／日期）＋ `target_folder`，**不讀取郵件內文、不含內容摘要**（較輕、較隱私）；分類者（人或 AI Agent）單憑這些標頭即可進行資料夾分類。
- **功能2 範圍**：列舉信箱**所有資料夾**作為參考清單。
- **無變動判定**：`target_folder` 空白或等於 `current_folder` ⇒ 不搬移。
- **目標資料夾**：不存在時於檢查報告標示，預設可自動建立（沿用既有 `ensure_folder`）。
- **破壞性把關**：功能3 預設先做檢查報告（dry-run 預覽），使用者明確確認後才真正搬移（符合憲法 Principle III）。
- **CSV 慣例**：UTF-8、含表頭、標準跳脫；預設檔名置於工作目錄（例如 `worksheet.csv`／`folders.csv`），可由使用者指定路徑。
- **CSV 欄位順序（固定）**：`uid, current_folder, target_folder, date, from, to, subject`（全英文表頭，利於 AI／LLM 與各 locale 試算表的穩健解析）。功能1 依此順序輸出、功能3 依此讀取，確保匯出與匯入一致（`target_folder` 緊接 `current_folder` 以便對照編輯）。
- **匯出覆寫**：功能1／2 匯出時，若目標檔已存在則**預設覆寫**（之後可再加另存／詢問選項）。
- **不重排列序**：功能3 依 CSV 列序逐列執行、不自動重排；某列來源在執行時已不存在（被同次較早列搬走）則列為該列失敗（見 FR-011）。
- **沿用既有基礎**：feature 001 的輸出安全與錯誤邊界、feature 002 的 `config.json` 設定與帳號驗證。
- **不在範圍內（本期）**：MailKeeper 內建 LLM 自動分類、以及把三功能自動串接成端到端自動流程（屬**未來階段**，待本期手動流程驗證可靠後再評估）；規則式自動分類（`organizer` 既有能力／Roadmap R5）；CSV 以外的匯出入格式；IMAP 以外後端；圖形介面。
