# Feature Specification: Bulk Move Efficiency & Idempotency — Phase 2 (P4 grouping + P3 no-redundant-SELECT + P2 batch move + C1 fallback idempotency)

**Feature Branch**: `007-bulk-move-efficiency`

**Created**: 2026-06-29

**Status**: Draft

**Input**: User description: 承接 feature 006（P1 存在性檢查最小化，v0.6.1）。本期把分類「執行搬移」階段的三層往返浪費（每封重複 SELECT、逐封 MOVE、來源夾交錯使前兩者失效）一併消除，並修正 fallback 搬移路徑的非冪等（COPY 後斷線重試會重複複本）。完成後進版 0.6.2。

> 對應 `doc/mailkeeper-performance-report-20260627.html` 的 **P4**（候選分組）+ **P3**（免重複 SELECT，= backlog C2）+ **P2**（批次 UID MOVE），並一併修 backlog **C1**（fallback 搬移冪等性）。P1 已於 006 完成；P5/P6/P7 仍延後。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 大量分類搬移大幅減少網路往返 (Priority: P1)

使用者用一份分類工作表把大量郵件（常達數百～數千封）搬到目標夾。今日每封搬移前都重新選取來源夾、且逐封搬移——對 500 封同來源夾的候選約需 1000 次網路往返（500 次多餘的重新選取 + 500 次逐封搬移），大信箱明顯緩慢。使用者期望同一來源夾的搬移**集中且批次**完成，大幅縮短時間。

**Why this priority**: 這是本期最大的效能價值，直接縮短大量搬移的耗時與被中斷的風險。需要三件事協同：先依來源夾分組（否則交錯來源夾會打斷後兩者）、同夾不重複選取、同夾同目標批次搬移。

**Independent Test**: 對「500 封同一來源夾→同一目標夾」與「來源夾交錯」兩種工作表執行搬移；以操作日誌驗證來源夾選取只發生一次、搬移命令次數由「每封一次」降為「每批一次」、候選依夾分組且順序具決定性，且最終搬移結果與逐封處理等價。

**Acceptance Scenarios**:

1. **Given** 500 封候選全為「同一來源夾→同一目標夾」，**When** 執行搬移，**Then** 該來源夾的重新選取次數為 1（不重複選取）、搬移命令次數為「每批一次」量級（非 500 次），且 500 封全部正確搬入目標夾。
2. **Given** 一份來源夾交錯（A、B、A、B…）的工作表，**When** 執行搬移，**Then** 候選以「先依來源夾、再依目標夾」的穩定順序處理（同夾相鄰）；對相同輸入兩次執行的處理順序一致（決定性）。
3. **Given** 任一工作表，**When** 比較優化前後，**Then** 最終搬移結果集合（哪些郵件進哪個目標夾）、每列成功／失敗、與呈現給使用者的逐列結果順序（依 CSV 工作表列序）皆等價。
4. **Given** 一個批次搬移中部分 UID 失敗（如其中一封來源已不存在），**When** 執行該批次，**Then** 系統能逐封歸因——成功者仍被搬移、失敗者如實回報，整體每列結果與「逐封處理」等價（批次不得讓某封失敗連坐其他封）。

### User Story 2 - 搬移在連線中斷／重試下不產生重複複本 (Priority: P2)

當伺服器不支援原子搬移而走後備路徑（複製→標記刪除→清除）時，若在「複製成功後、清除前」發生連線中斷，今日的透明重連重試會重新複製 → 目標夾出現**重複複本**。使用者期望即使中途斷線並自動重試，搬移結果仍正確：目標夾每封只有一份、來源正確移除。

**Why this priority**: 這是資料正確性問題（backlog C1）。批次／重連使中斷窗口更需要被正確處理；雖非資料遺失，重複複本會污染使用者信箱、破壞信任。

**Independent Test**: 在後備搬移路徑注入「複製成功後、清除前」中斷，觸發透明重連重試；驗證目標夾無重複複本、來源正確移除、他人已標記刪除的郵件不被波及。

**Acceptance Scenarios**:

1. **Given** 後備搬移路徑於「複製成功、清除前」中斷並透明重連重試，**When** 重試該封搬移，**Then** 目標夾僅有一份複本（不重複）、來源該封正確移除。
2. **Given** 搬移過程（主路徑或後備）中途連線中斷／權杖過期，**When** 透明重連後續完，**Then** 從中斷處繼續、不重複搬移、不遺漏。
3. **Given** 來源夾中另有他人標記刪除的郵件，**When** 任何搬移／清除發生，**Then** 該他人郵件全程不被波及（不被連坐清除）。

### User Story 3 - 結果與安全等價現況（無回歸）(Priority: P3)

使用者期望本期只改「搬移如何送命令」，不改變任何使用者可見的結果與安全保證：搬移結果、逐列回報、預設 dry-run（未確認不動信箱）、進度顯示、不外洩任何祕密，全部維持。

**Why this priority**: 回歸防線——分組／批次／重連改動風險集中於搬移路徑（唯一真接協定、最高風險），必須守住結果等價與破壞性動作安全。

**Independent Test**: 對同一工作表執行搬移，逐列結果與順序、dry-run 預覽行為與優化前一致；確認破壞性動作仍預設 dry-run、輸出無 token；他人已標刪郵件不被波及。

**Acceptance Scenarios**:

1. **Given** 任何分類操作未加執行旗標，**When** 執行，**Then** 維持 dry-run（不變更信箱），輸出／錯誤訊息不含 token 或祕密。
2. **Given** 確認後執行，**When** 搬移完成，**Then** 結果集合、逐列成功／失敗與順序與優化前等價；破壞性動作維持「複製成功才標刪、以 UID 限定清除該封」的安全鐵則。

---

### Edge Cases

- **同夾多目標**：同一來源夾、不同目標夾的候選 → 分組以 (來源,目標) 為鍵；各 (來源,目標) 群各自批次。
- **批次部分失敗**：批次中某些 UID 失敗（來源已不存在／伺服器拒絕）→ 退回逐封歸因，成功者仍搬、失敗者如實回報。
- **後備路徑重試重複窗口**：複製成功後、清除前中斷重試 → 冪等處理（偵測來源 UID 狀態或目標既有複本）避免重複複本。
- **搬移中途斷線／token 過期**：透明重連後從中斷處續完；已搬走者重試為 no-op（不重複）。
- **來源 UID 執行時已不存在**：如實回報該列失敗（不崩潰、不連坐批次其他封）。
- **空候選 / 單封**：批次退化為單封；行為與逐封等價。
- **連續失敗早停**：分組／批次下「連續失敗早停」的判定方式需明確（見 Assumptions 之過渡決策，待 `/speckit.clarify` 確認）。
- **他人已標記刪除郵件**：任何清除絕不波及之（UID 限定清除）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 分類執行 MUST 先將候選依「來源資料夾（再依目標資料夾）」穩定分組／排序，使同一 (來源,目標) 的候選相鄰、處理順序具決定性、可重現。
- **FR-002**: 呈現給使用者的逐列搬移結果 MUST 依原輸入工作表列序排列（內部分組處理、輸出還原列序；沿用 feature 006 的呈現順序決策）。
- **FR-003**: 同一來源夾連續操作時 MUST NOT 重複選取該夾——僅在「未選／資料夾不同／讀寫模式不同」時才重新選取；連線建立／重連後選取狀態 MUST 重置（重連後必重新選取）。
- **FR-004**: 同一 (來源夾→目標夾) 的多封 MUST 能以單一批次搬移（UID 集合）一次完成，將「每封一次往返」收斂為「每批一次往返」。
- **FR-005**: 批次搬移於「批次中部分 UID 失敗」時 MUST 退回逐封處理以精確歸因——成功者仍搬移、失敗者如實回報；單封失敗 MUST NOT 連坐同批其他封。最終每列結果 MUST 與逐封處理等價。
- **FR-006**: 後備搬移路徑（伺服器不支援原子搬移）MUST 為冪等：於「複製成功後、清除前」中斷並重試時，MUST NOT 在目標夾產生重複複本（重試前偵測來源 UID 狀態或目標既有複本）；來源該封 MUST 正確移除。
- **FR-007**: 搬移過程（主／後備路徑）連線中斷或權杖過期時 MUST 透明重連並從中斷處續完，MUST NOT 重複搬移或遺漏；已搬走者之重試 MUST 為 no-op。
- **FR-008**: 對相同輸入工作表，優化前後的最終搬移結果集合（被搬郵件與其目標夾、每列成功／失敗）MUST 完全一致。
- **FR-009**: 系統在搬移全程（含恢復／重連期間）MUST 持續顯示進度，且進度回報語意不退化。
- **FR-010**: 破壞性動作 MUST 維持既有安全鐵則：複製成功才標記刪除；以 UID 限定清除該封（不波及他人已標記刪除郵件）；MUST 維持 dry-run 預設（未明確確認不變更信箱）。
- **FR-011**: 批次搬移、選取追蹤、後備冪等所需的能力 MUST 以後端中立方式提供（透過既有後端抽象，由各後端實作）；上層分類／CLI MUST NOT 特例化任一後端，協定細節 MUST NOT 跨越後端邊界外洩。
- **FR-012**: 本變更 MUST NOT 記錄或輸出任何存取權杖／祕密，且 MUST NOT 變更 OAuth 授權範圍。
- **FR-013**: 「連續失敗早停」在分組／批次下的判定行為 MUST 明確定義（過渡預設見 Assumptions）；其最終定義由 `/speckit.clarify` 決定後落入本需求。

### Key Entities *(include if feature involves data)*

- **搬移群組（Move Group）**：以 (來源資料夾, 目標資料夾) 為鍵聚合的候選 UID 清單；批次搬移與分組處理的單位。
- **逐列搬移結果（Per-Row Move Result）**：每列的成功／失敗與原因；呈現順序依原 CSV 工作表列序（FR-002）。
- **目前選取狀態（Selected Mailbox State）**：客戶端目前選取的 (資料夾, 讀寫模式)；用於免重複選取（FR-003），連線／重連後重置。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 對「500 封同一來源夾」的搬移，來源夾重新選取次數為 **1**（同夾同模式零重複選取，可由模擬器 `redundant_selects()==0` 與指令日誌驗證）。
- **SC-002**: 對「500 封同一 (來源→目標)」的搬移，搬移命令的網路往返由「每封一次」降為「每批一次」量級（**N → ⌈N/批⌉**，可由指令日誌的搬移命令次數驗證）。
- **SC-003**: 對來源夾交錯的工作表，候選以「同夾相鄰」的決定性順序處理（重複執行順序一致）；最終搬移結果集合、每列成功／失敗、逐列輸出順序（依 CSV 列序）與優化前 **100%** 等價。
- **SC-004**: 批次中部分 UID 失敗時，逐封歸因正確——成功者全數搬移、失敗者全數如實回報，與逐封處理結果 **100%** 等價。
- **SC-005**: 後備路徑於「複製後、清除前中斷重試」情境，目標夾複本數 **正好 1**（不重複）、來源正確移除（feature 006 的 C1 xfail 測試屆時自動 xpass、移除 marker）。
- **SC-006**: 搬移中途連線中斷／權杖過期 → 透明重連後全部完成、**0 重複、0 遺漏**、零人工介入。
- **SC-007**: 任何搬移／清除全程，他人已標記刪除郵件被波及次數為 **0**（不連坐清除）。
- **SC-008**: 未加執行旗標時 **100%** 維持 dry-run（不變更信箱）；輸出無任何 token／祕密。
- **SC-009**: 搬移全程（含恢復／重連期間）皆有進度回饋，無「無回饋、看似當機」的等待區段。

## Assumptions

- 後端能以單一批次搬移多個 UID（IMAP 提供 UID 集合／範圍的搬移與複製能力；離線測試用 `FakeBackend` 亦可提供等價批次能力與「來源 UID 狀態查詢」以驗冪等）。
- 批次大小：以「整個同 (來源,目標) 群為一批」為基準，必要時以合理上限分批（過渡預設；確切上限／是否納入設定可於 `/speckit.clarify` 細化）。
- 進度：批次搬移時進度以「每批完成 += 該批封數」推進至總數（過渡預設；確切粒度於 `/speckit.clarify` 確認，沿用 006 對進度不退化的精神）。
- **早停（過渡決策，待 `/speckit.clarify` 確認）**：分組／批次會改變「原列序下連續失敗」的計數，故過渡上以「連線層級失敗（疑似斷線）」作為提前停止依據，而非單純的列序連續計數；最終語意由 clarify 決定。
- fallback 冪等以「重試前偵測來源 UID 是否仍在／是否已標記刪除」或「目標夾偵測既有複本」實現；UID 為跨流程穩定識別。
- 本期不改 dry-run 預設、不換後端、不動 OAuth；不含 P5/P6/P7。
