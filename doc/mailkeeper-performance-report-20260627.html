一句話心智模型：IMAP 是同步協定——送一個命令、等一個回應，就是「一次往返（round-trip）」。 在大信箱上，本地 CPU 幾乎不是瓶頸，往返次數才是。所有優化的目標只有一個：把往返次數降下來，且不犧牲正確性與安全。
浪費的往返／白工 必要且高效的往返 命令／回應 新發現 已在 backlog
錨定情景1 萬封 · 搬 500 封 📌 狀態追蹤已完成／未實作 P1 · 最高存在性檢查過度抓取 P2批次 UID MOVE P3 · backlog C2重複 SELECT P4 · 前置候選分組排序 P5重連整夾重抓 P6批量大小可調 P7解析微優化 落地順序與約束

錨定情景：把這些抽象優化變成「你感受得到的數字」

整份報告共用同一個具體情景，方便對照每項優化的貢獻：

📥 INBOX 有 10,000 封郵件；你準備一份分類工作表（CSV），其中 500 封要從 INBOX 搬到 Work 夾。
流程：classify 先產檢查報告（dry-run）確認來源郵件存在，再執行搬移。

P1 分類存在性檢查「過度抓取」整封標頭 新發現 影響：最高 工：中 風險：低

問題

分類報告（build_report）要判斷「CSV 裡每一列的來源 UID 是否真的存在於來源夾」。它取得「來源夾現存 UID 集合」的方式是—— 呼叫 list_headers(folder)，這會對整夾每一封下 FETCH (UID BODY.PEEK[HEADER.FIELDS (SUBJECT FROM TO DATE)])， 逐封跑 email 解析 + 4 次 _decode()……然後只留下 h.uid，其餘全部丟掉。

  • classifier.py:78 — {h.uid for h in backend.list_headers(folder)}（只要 uid）
  • classifier.py:98-111 — build_report 全程從不讀 subject／from
  • 真相：要的只是「UID 清單」，但 UID SEARCH ALL 一次就能拿到（且它已經在 _list_headers_impl:294 被呼叫過，只是之後又多此一舉地 FETCH 全表頭）

情景演示（來源夾 = 10,000 封的 INBOX）

Before · 為了 UID 卻下載全表頭

1→SELECT INBOX
2→UID SEARCH ALL← 拿到 1 萬個 UID（其實已足夠！）
3→UID FETCH 1..50 (HEADER.FIELDS…)
…→…再 199 批…每批回 50 封完整表頭
202→UID FETCH 9951..10000
≈ 202次往返+ 10,000次解析（全部丟棄）

After · 只問「有哪些 UID」

1→SELECT INBOX
2→UID SEARCH ALL← 1 萬個 UID，完成
≈ 2次往返+ 0解析浪費

同一份 UID 集合經 ClassifyCache 給「執行階段」重用——存在性檢查零額外成本。

改法

在後端 seam 介面 MailBackend 加一個輕量方法：

class MailBackend(Protocol): def list_uids(self, folder: str) -> set[str]: ... # 只回 UID，不抓內容 # imap_client 實作 = 一次 SELECT + 一次 UID SEARCH ALL： def list_uids(self, folder): self._conn.select(folder, readonly=True) typ, data = self._conn.uid("search", None, "ALL") return set(data[0].split().decode...) if typ=="OK" else set()

分類路徑（_source_uids）改呼叫 list_uids；list_headers（功能1 匯出／功能2 列標題真的需要表頭）維持不變。

⚠ 跨 seam 新方法 → 須同步：FakeBackend 實作 + IMAP 模擬器引擎以 assert_all_fetches_request_uid()／log 驗證「分類報告不再對整夾 FETCH」。

P2 搬移是「每封一次往返」，未用 UID 範圍批次 新發現 影響：高 工：高 風險：中

問題

execute 對每個候選呼叫一次 backend.move()（classifier.py:170），底層 _move_impl 對單一 UID 下 UID MOVE（imap_client.py:366）。IMAP 其實支援一次搬一整組：UID MOVE 1,2,3,7:9 Work。

情景演示（搬 500 封，已假設同夾、且已用 P3 免去重複 SELECT）

Before · 500 封 = 500 次 MOVE

1→UID MOVE 10 Work
2→UID MOVE 11 Work
…→…498 次…
500→UID MOVE 783 Work
500次往返

After · 批次 100／組 = 5 次 MOVE

1→UID MOVE 10:120,… Work100 封一組
…→… 3 組 …
5→UID MOVE … Work
5次往返

改法 & 取捨

  • 把候選依 (current_folder → target_folder) 分組，每組以 UID 集合語法批次 UID MOVE。
  • 取捨① 進度粒度：進度條從「逐封」變「逐組」，需把進度改成「每組完成 += 該組封數」。
  • 取捨② 錯誤歸因：一組裡某幾封失敗時，伺服器不會逐封告訴你誰失敗 → 策略：批次失敗就退回該組「逐封」重試，定位到具體 UID。
  • 取捨③ 與冪等性糾纏：批次和 backlog 的 C1 fallback 冪等性 互相影響，建議同一個 feature 一起設計。
P3 搬移迴圈對「同一個夾」重複 SELECT backlog C2 影響：中 工：低 風險：低

問題

_move_impl 每一次都先 self._conn.select(mailbox)（imap_client.py:365），但 client 並沒有記住「我現在選的是哪個夾」。於是搬 500 封同來源 = 多打了 ~499 次完全多餘的 SELECT（每次 SELECT 還是一次往返，伺服器又重報一次 EXISTS／FLAGS）。

情景演示（搬 500 封同夾）

Before · SELECT + MOVE 反覆

1→SELECT INBOX多餘（已選）
2→UID MOVE 10 Work
3→SELECT INBOX又多餘
4→UID MOVE 11 Work
…→… ×500 …
≈ 1000次往返（500 是白工）

After · 只在切夾時 SELECT

1→SELECT INBOX只此一次
2→UID MOVE 10 Work
3→UID MOVE 11 Work
…→… 只剩 MOVE …
≈ 501次往返

改法

OutlookIMAPClient 記住目前選取的 (mailbox, readonly)，加 _ensure_selected()：只在「未選 / 夾不同 / 模式不同」時才真的 SELECT。

def _ensure_selected(self, mailbox, readonly): if self._selected == (mailbox, readonly): return # 已選同夾同模式 → 跳過 SELECT self._conn.select(mailbox, readonly=readonly) self._selected = (mailbox, readonly) # connect() / _reconnect() 要把 self._selected = None（連線換了，選取失效）

⚠ 鐵則：重連後選取一定失效，必須重置 _selected 才能重新 SELECT；引擎測試 redundant_selects()==0 把關。

P4 候選未依資料夾分組排序 —— 它讓 P2／P3 失效 新發現 影響：放大器 工：極低 風險：零

問題

execute 照 CSV 原始列序逐列搬（classifier.py:160）。如果使用者的工作表來源夾是交錯的（A、B、A、B…），那麼：

  • P3 的「記住已選夾」會一直被打斷（選 A→選 B→又選 A…每次都得重 SELECT）。
  • P2 的「同夾批次 MOVE」也湊不成一組（相鄰兩列不同夾就無法合併）。

情景演示（4 列：INBOX、Promo、INBOX、Promo）

Before · 交錯 → 選取狂翻

SELECT INBOX → MOVE SELECT Promo → MOVE 切夾 SELECT INBOX → MOVE 切回 SELECT Promo → MOVE 又切 = 8 次往返、0 封能批次

After · 先排序 → 同夾相鄰

sort by (current → target) SELECT INBOX → MOVE 1,3 批次 SELECT Promo → MOVE 2,4 批次 = 4 次往返、可批次

改法

執行前一行純本地排序，零網路、零風險：cands.sort(key=lambda it:(it.row.current_folder, it.row.target_folder))。

💡 P4 是 P2／P3 的「前置乘數」——它本身省不了多少，但沒有它，P2／P3 在真實交錯資料上會大打折扣。所以建議第一個做。
P5 list_headers 重連會「整夾重抓」，是浪費放大器 新發現 影響：中（尾端斷線時很痛） 工：中高 風險：中

問題

_with_reconnect 包住整個 _list_headers_impl（imap_client.py:288，docstring 明載「整批重抓」）。對功能1 匯出／功能2 列標題這種真的需要全表頭的路徑（P1 救不到），一旦在尾端斷線，前面抓好的全部作廢、從頭再來。

情景演示（匯出 10,000 封，在第 199/200 批 token 過期）

Before: 批1 ✓ 批2 ✓ … 批199 ✓ ← 斷線！ 重連後 → 批1 批2 … 批200 (整整 199 批白抓) After : 批1 ✓ 批2 ✓ … 批199 ✓ ← 斷線 重連後 → 只續抓 批200 (跳過已取得的 UID)

改法

  • 把「已取得的 UID」記為集合；重連後重 SELECT + UID SEARCH ALL，只 FETCH 還沒抓到的 UID 續完。
  • 唯讀、以 UID 為鍵 → 續抓冪等且安全。
  • 鐵則：重連後若 UIDVALIDITY 變了（信箱被重建），代表 UID 空間整個換了 → 必須放棄續抓、整批重來（這時重抓才是正確的）。

套用 P1 後，分類路徑已改用單次 SEARCH（重抓很便宜），所以 P5 的價值主要落在匯出大信箱。優先序因此排在 P1/P3 之後。

P6 FETCH 批量大小寫死、不可調 新發現 影響：低 工：低 風險：低

問題 & 改法

_FETCH_BATCH = 50 是模組常數（imap_client.py:39），不在 config.json 可覆寫範圍（R7 只開放了失敗門檻／退避秒數）。Outlook 可吞更大的 UID 集合；把批量調大可減往返。

批量 50（現況）

10,000 / 50 = 200 批 FETCH

批量 200（可調）

10,000 / 200 = 50 批 FETCH
  • 把批量納入 config.json（沿用既有 config_store 的「無效則退回安全預設」模式）。
  • 取捨：批量越大 → 往返越少，但進度更新越粗、單次回應記憶體越大。給個合理上限即可。
P7 本地解析微優化（誠實標註：邊際效益） 新發現 影響：低 工：低 風險：低
  • 表頭解析：email.message_from_bytes（imap_client.py:326）走的是「全訊息」解析機制；我們只 PEEK 表頭欄位，可改 email.parser.BytesHeaderParser 省一點 CPU。（屬微優化，往返沒變，只有超大信箱才感受得到。）
  • 誠實澄清誤報：_decode_mutf7 雖被知識圖譜複雜度指標標為「迴圈內線性掃描 / 配置」熱點，但它只在列資料夾（LIST）時跑、非每封郵件——實務上不是瓶頸，不需要動。把它列出來是為了說明「指標會誤報，仍需讀碼判斷」。

已經做對、不需要再優化的部分

機制現況（已是好的）
連線重用單一連線貫穿全程，沒有逐操作重連的浪費。
報告 ↔ 執行 不二次掃描ClassifyCache.source_uids 讓「資料夾清單」與「來源夾 UID 集合」各只讀一次（FR-007），執行階段重用、present.discard 維持快取鮮活。P1 是在此基礎上「連那一次也從 FETCH 降成 SEARCH」。

建議落地順序與流程約束

①P4 排序（零風險前置）
②P1 classify 最大刀（FETCH→SEARCH）
③P3／C2 搬移砍半（_ensure_selected）
④P2 批次 MOVE（含 C1 冪等性一起設計）
⑤P5 可續傳重連（匯出大信箱）
⑥P6 批量可調
程序約束（CLAUDE.md §2 / §4 / §7）：
  • 以上皆產品碼變更 → 開新 feature 分支 NNN-name，走 /speckit.specify → clarify → plan → tasks → analyze → implement → SR，升版 + 更新 CHANGELOG。
  • 任何跨 seam 改動必須用 IMAP 模擬器引擎實測，以 loop_report() / redundant_selects() / assert_all_fetches_request_uid() 驗證「往返真的變少、且沒破壞 UID 索取與破壞性動作安全」。
  • P1／P2／P3／P5 與 backlog 的 R7-followup C1（fallback move 冪等性） C2（重 SELECT） 高度相關——若要動搬移路徑，建議把 C1、C2 併入同一 feature，避免反覆改 _move_impl。
