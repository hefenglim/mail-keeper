01 · Overview

專案概覽

MailKeeper 是一支命令列工具，登入個人 Outlook.com 信箱，列出收件匣所有郵件的標題，並依使用者定義的規則自動整理（搬移資料夾、標記已讀、加旗標）。

整個設計只圍繞一個核心想法：郵件存取層是可替換的。所有 IMAP 協定細節都封裝在單一模組，整理邏輯只依賴一個抽象介面 MailBackend。因此未來要把底層從 IMAP 換成 Microsoft Graph API，整理層一行都不必動。Claude Code 接手時，請把這條界線視為不可破壞的約束。

關鍵前提 · 不可繞過 Outlook.com 已停用 Basic Auth（帳號＋密碼直連）。IMAP 一律走 OAuth2 / XOAUTH2，必須先在 Azure / Microsoft Entra 註冊應用程式取得 CLIENT_ID。任何「改用帳密登入」的修改都會失效，請勿嘗試。

04 · Contract

MailBackend 契約

這是整個專案的樞紐。任何郵件後端（IMAP、Graph、未來的測試替身）都必須提供以下方法。簽章定義於 organizer.py，請勿隨意更動；若要擴充，採新增方法而非破壞既有簽章。

# organizer.py
class MailBackend(Protocol):
  def list_inbox_headers(self, mailbox: str = "INBOX") -> list[MailHeader]
  def ensure_folder(self, folder: str) -> None
  def move(self, uid: str, dest_folder: str, mailbox: str = "INBOX") -> None
  def mark_read(self, uid: str, mailbox: str = "INBOX") -> None
  def flag(self, uid: str, mailbox: str = "INBOX") -> None
方法用途實作要點
list_inbox_headers列出指定信箱所有郵件標題回傳 MailHeader 串列；IMAP 版只抓 HEADER.FIELDS (SUBJECT FROM DATE)
ensure_folder確保目標資料夾存在已存在可忽略錯誤（IMAP 回 NO）
move把郵件搬到資料夾IMAP 用 UID MOVE，失敗時 copy + \Deleted + expunge
mark_read標記為已讀IMAP +FLAGS (\Seen)
flag加上旗標IMAP +FLAGS (\Flagged)

06 · Domain Model

領域模型

上層只看得到這兩個資料類別，看不到任何 IMAP response，這是隔離得以成立的關鍵。

@dataclass(frozen=True)
class MailHeader:
    uid: str          # IMAP UID（後端內部識別）
    subject: str      # 已解碼標題
    sender: str       # From
    date: str         # Date 原字串

@dataclass
class Rule:
    name: str
    match: Callable[[MailHeader], bool]   # 命中條件
    dest_folder: str | None = None      # 要搬去的資料夾
    mark_read: bool = False
    flag: bool = False
設計取捨 uid 是 IMAP 特有概念。若日後改用 Graph（用 message id），可保留 uid 欄位語意為「後端不透明識別字」，介面不變即可相容。

07 · One-time Setup

環境設定（Azure 註冊）

跑起來前必做一次，否則無法登入：

  • Microsoft Entra 系統管理中心 → 應用程式註冊 → 新增註冊
  • 支援帳戶類型選 「個人 Microsoft 帳戶」（或含工作/學校帳戶）
  • 不需填 redirect URI（採 device code flow）
  • 複製 應用程式（用戶端）識別碼 → 填入 config.py 的 CLIENT_ID
  • 驗證 → 允許公用用戶端流程 → 是
  • API 權限：授予委派權限 IMAP.AccessAsUser.All（範圍 outlook.office.com）

08 · Build & Run

建置與執行

# 開發模式安裝（可邊改邊跑）
pip install -e .

# 填好 config.py 的 CLIENT_ID 後執行
mailkeeper
# 或
python -m mailkeeper

# 打包
python -m build        # 產生 dist/*.whl 與 *.tar.gz

首次執行會印出「開啟網址並輸入代碼」的 device code 提示，登入後 token 快取，之後免登入。整理動作預設 dry_run=True 只顯示不變更；確認規則無誤後，把 cli.py 內 organizer.run(dry_run=False) 打開才會真的動信箱。

09 · Conventions

開發規範

  • 隔離不可破：imaplib 僅限 imap_client.py；上層只依賴 MailBackend 與 MailHeader。
  • 新供應商＝新類別：實作 MailBackend 五個方法即可，不改 organizer/cli。
  • 語言慣例：識別字、型別、commit 用英文；說明性註解可用繁中（人面向）。
  • 型別註解：全檔 from __future__ import annotations；維持 mypy 友善。
  • 版本紀律：每次發版更新 CHANGELOG.md，版本標題使用真實交付日期，不要造假日期。
  • 機密：token_cache.bin 含有效憑證，已列入 .gitignore，絕不提交。
  • 破壞性動作預設關閉：涉及搬移/刪除的新功能，預設走 dry-run，由使用者明確開啟。

10 · Testing

測試策略

整理層可完全離線測試 —— 用實作 MailBackend 的假後端注入 MailOrganizer，不需真的連 IMAP。此模式已驗證可行，建議 Claude Code 以此為 pytest 基礎：

class FakeBackend:
    def __init__(self): self.actions = []
    def list_inbox_headers(self, mailbox="INBOX"):
        return [MailHeader("1", "Weekly Newsletter", "news@x.com", "Mon")]
    def ensure_folder(self, f): self.actions.append(("folder", f))
    def move(self, uid, dest, mailbox="INBOX"): self.actions.append(("move", uid, dest))
    def mark_read(self, uid, mailbox="INBOX"): self.actions.append(("read", uid))
    def flag(self, uid, mailbox="INBOX"): self.actions.append(("flag", uid))

優先補的測試：規則比對器（from_contains/subject_contains）、_decode() 對 encoded-word 標題的解碼、MailOrganizer.run() 在 dry-run 與實跑下產生的動作序列。

11 · Roadmap

後續工作

建議的接續順序，優先級越高越該先做。每項完成後請更新 CHANGELOG.md 並升版。

ID任務優先重點
R1CLI 參數化高加 argparse：--version、--run（關掉 dry-run）、--list（只列標題）、--mailbox
R2pytest 測試套件高以 FakeBackend 為基礎，涵蓋規則、解碼、run() 動作序列
R3Microsoft Graph 後端中新增 graph_client.py 實作 MailBackend；Graph 原生支援資料夾/分類，更適合 Outlook 專用情境
R4設定外部化中CLIENT_ID / EMAIL 改讀環境變數或 .env；偵測未填的佔位字串並報錯
R5規則外部化中規則改由 YAML/JSON 載入，免改程式即可調整整理需求
R6logging 取代 print低結構化輸出、可調等級
R7大量信箱效能低UID 範圍批次 fetch、分頁；斷線重連與 token 過期重試
待人類決策 若這支工具只服務 Outlook，R3 的 Graph 後端整理體驗更佳（原生資料夾搬移、分類標籤、伺服器端篩選）；若要跨供應商通用（Gmail/Yahoo），則維持 IMAP 為主。動工 R3 前請先向使用者確認方向。
